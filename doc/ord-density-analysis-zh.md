# V6.4.3: ord 密度测量与决策

> 前置阅读：`vector-db-design-zh.md`（ord 模型）、`format-zh.md`（posting 压缩格式）
> 状态：设计中（不包含实现代码）

## 1. 动机

Bitcask ord 模型采用「单调递增、永不复用」策略。随着删除和压缩，posting list 中会出现 ord 空隙（gap）。若空隙过大，会导致 FOR（Frame-of-Reference）压缩效率下降，从而增加磁盘占用和 I/O 开销。

本设计旨在：
1. 量化 ord gap 分布（1M/10M 文档规模）
2. 分析 FOR 压缩在不同 gap 下的性能影响
3. 决策是否需要 ord 重编号（ord renumbering）

## 2. Ord 模型回顾

### 2.1 Ord 分配语义

```
insert doc A → ord=0
insert doc B → ord=1
insert doc C → ord=2
delete doc B → ord=1 标记为 dead（但仍存在 posting list 中）
compact → ord=1 从所有 posting list 移除
```

**关键不变式**：
- ord 单调递增分配
- ord **永不复用**（即使删除后，新文档分配新 ord）
- compact 仅从 posting list 移除 dead ords，不重编号

### 2.2 Gap 定义

Posting list 中相邻两个 ord 的差值即为 gap：

```
posting list: [ord=0, ord=2, ord=5]
gap1 = 2 - 0 = 2
gap2 = 5 - 2 = 3
average_gap = (2 + 3) / 2 = 2.5
density = 1 / average_gap = 40%
```

理想情况（无删除）：所有 gap = 1，density = 100%

## 3. Gap 分布分析

### 3.1 理论模型

假设总文档数 N，删除率 p，删除后 compact：

- 存活文档数 = N × (1 - p)
- ord 空间大小 = N（ord 永不复用）
- 平均 gap = N / (N × (1 - p)) = 1 / (1 - p)
- density = 1 / average_gap = 1 - p

**关键观察**：gap 仅取决于删除率 p，与绝对规模无关。

### 3.2 典型场景推演

| 删除率 p | 平均 gap | 密度 | 备注 |
|---|---|---|---|
| 0% | 1.0 | 100% | 纯追加，无删除 |
| 10% | 1.11 | 90% | 常见场景 |
| 30% | 1.43 | 70% | 中度删除 |
| 50% | 2.0 | 50% | 重度删除 |
| 70% | 3.33 | 30% | 极端场景 |
| 90% | 10.0 | 10% | 数据归档场景 |

### 3.3 实测数据（待 V6.4.3 验收）

**实验设计**：
1. 构造 1M 文档集合，插入 1M 文档
2. 随机删除 p% 文档（p = 0%, 10%, 30%, 50%, 70%, 90%）
3. 触发 compact，测量 posting list gap 分布
4. 重复 10 次，取平均 gap 和密度

**预期结果**：与理论模型一致，gap = 1 / (1 - p)

## 4. FOR 压缩影响分析

### 4.1 FOR 编码原理

FOR（Frame-of-Reference）用于压缩 posting list 中的 ord 序列：

```
ord list: [100, 102, 105, 110]
base = 100
deltas: [0, 2, 5, 10]
bit-width = ceil(log2(max_delta + 1)) = ceil(log2(11)) = 4 bits
compressed_size = len(deltas) × bit-width / 8 = 4 × 4 / 8 = 2 bytes
```

**关键公式**：
- `max_delta = max(ords) - min(ords)`
- `bit-width = ceil(log2(max_delta + 1))`
- `compressed_size = len × bit-width / 8`

### 4.2 不同 gap 下的压缩开销

以 128 个 ord 为一个 block（常见分块大小）：

| Gap（平均） | 密度 | Max Delta | Bit-width | 压缩后大小 | 相对 gap=1 增幅 |
|---|---|---|---|---|---|
| 1.0 | 100% | 127 | 7 | 112B | 基准 |
| 1.11 | 90% | 140 | 8 | 128B | +14% |
| 1.43 | 70% | 181 | 8 | 128B | +14% |
| 2.0 | 50% | 254 | 8 | 128B | +14% |
| 3.33 | 30% | 422 | 9 | 144B | +29% |
| 10.0 | 10% | 1270 | 11 | 176B | +57% |

**观察**：
- gap 从 1.0 增至 2.0（删除率 0% → 50%），bit-width 从 7 → 8，压缩开销仅 +14%
- gap > 3.33（删除率 >70%），压缩开销显著增加（+29% ~ +57%）

### 4.3 存储开销推演

假设 1M 文档，平均 posting list 长度 1000 ord，FOR block size 128：

| 删除率 p | Gap | FOR 块数 | 每块大小 | 总大小 | 相对 p=0% 增幅 |
|---|---|---|---|---|---|
| 0% | 1.0 | ~7.8M | 112B | ~874MB | 基准 |
| 10% | 1.11 | ~7.0M | 128B | ~896MB | +2.5% |
| 30% | 1.43 | ~5.5M | 128B | ~704MB | -19%（文档数减少主导） |
| 50% | 2.0 | ~3.9M | 128B | ~500MB | -43% |
| 70% | 3.33 | ~2.3M | 144B | ~331MB | -62% |
| 90% | 10.0 | ~0.8M | 176B | ~141MB | -84% |

**关键结论**：
- 删除率 ≤ 50% 时，FOR 压缩开销增加（+14%）被文档数减少（-50%）完全抵消，总存储仍显著降低
- 即使 gap=2.0（删除率 50%），压缩效率仍可接受（+14% 开销）

## 5. 决策：正式取消 ord 重编号

### 5.1 决策依据

1. **删除率 ≤ 50% 时，FOR 开销仅 +14%**
   - 典型 KV 工作负载删除率 < 30%（数据保留需求）
   - 14% 存储开销换 ord 重编号的复杂度不划算

2. **ord 重编号成本高昂**
   - 需修改快照（snapshot）、WAL、活索引（live index）、KeyDir 四个组件
   - 重编号期间需要全局锁或复杂协调机制，影响写入 QPS
   - 破坏「永不复用」不变式，增加语义复杂度

3. **极端场景（删除率 > 80%）可规避**
   - 归档场景（删除率 > 80%）通常不会与检索混用
   - 可通过「分集合」策略：热集合（低删除率）+ 冷集合（归档）

### 5.2 决策

**正式取消 ord 重编号，延后至 V8 或更晚版本**。

### 5.3 阈值记录

若未来删除率 > 80% 成为常见场景（如频繁 TTL 过期），可重新考虑 ord 重编号。此时：
- FOR 压缩开销 > +57%（gap > 10）
- ord 重编号的收益开始大于成本
